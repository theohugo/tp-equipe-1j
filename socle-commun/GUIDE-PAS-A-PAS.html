Guide pas-a-pas - Construire votre RAG, etape par etape

Ce guide vous accompagne du clone jusqu'a la demo. Il explique chaque etape : ce que vous faites, pourquoi, comment vous y prendre, un indice de code, et comment verifier que ca marche avant de passer a la suite. Lisez-le dans l'ordre. Gardez le GLOSSAIRE.md ouvert a cote pour les mots inconnus.

🔷 Regle d'or : on avance etage par etage et on verifie a chaque fois. Ne codez jamais 3 etapes d'un coup sans tester. Un RAG qui marche a 80 % et qu'on a verifie vaut mieux qu'un RAG complet qui ne demarre pas.

Comprendre le RAG en 5 minutes (a lire en equipe avant de coder)

RAG = Retrieval-Augmented Generation. En francais : on retrouve les bons passages dans vos documents, puis on demande a une IA de rediger une reponse a partir uniquement de ces passages.

Analogie simple : imaginez un etudiant a un examen "open book". On lui pose une question. Il ne repond pas de tete (il halluciniinerait) : d'abord il cherche les bonnes pages dans ses classeurs (c'est le retrieval), ensuite il redige la reponse en citant ces pages (c'est la generation). Votre RAG fait pareil, automatiquement.

Pourquoi pas juste demander a ChatGPT ? Parce que le LLM ne connait pas vos documents internes et qu'il invente quand il ne sait pas. Le RAG lui fournit la matiere et l'oblige a citer ses sources.

Etape 1 (R1) - Ingestion : decouper les documents

Ce que vous faites : lire chaque document, en extraire le texte, le decouper en morceaux (chunks) et attacher des metadonnees (d'ou vient le morceau).

Pourquoi decouper ? Un LLM ne peut pas avaler un PDF de 80 pages d'un coup, et on veut retrouver le passage precis qui repond, pas tout le document. On coupe donc en morceaux de ~800 caracteres.

Analogie : on ne photocopie pas tout le manuel pour repondre a une question ; on marque les paragraphes utiles avec des post-it. Un chunk = un paragraphe avec son post-it (la source).

Comment s'y prendre

Le fichier app/ingest.py est deja amorce. Votre travail :

  1. Verifier que l'extraction marche pour vos formats (PDF pour C, JSON/HTML pour B, divers pour A).
  2. Regler CHUNK_SIZE et CHUNK_OVERLAP (dans .env). Valeurs de depart : 800 / 120.
  3. Enrichir les metadonnees (la source au minimum ; voir les indices de votre projet).

Indice (le coeur du decoupage, deja dans le code) :

from langchain_text_splitters import RecursiveCharacterTextSplitter
splitter = RecursiveCharacterTextSplitter(chunk_size=800, chunk_overlap=120)
chunks = splitter.split_text(texte)   # liste de morceaux qui se recouvrent un peu

Le recouvrement (overlap) evite de couper une phrase importante en deux : deux chunks voisins partagent 120 caracteres.

Comment verifier

docker compose run --rm api python -m app.ingest

Output attendu (la forme) :

[ingest] 34 documents -> 612 chunks dans corpus/chunks.jsonl

Puis regardez le resultat :

head -n 1 corpus/chunks.jsonl   # doit afficher un JSON avec text + metadata (source, position)

🟢 OK si : chunks.jsonl existe, contient des chunks lisibles (pas de charabia) et chaque chunk a une source.

Erreurs frequentes a cette etape :

🟨 Symptome Cause probable Correction
chunks.jsonl vide aucun fichier dans corpus/ relancer fetch_corpus.sh, verifier corpus/seed/
texte illisible (caracteres bizarres) mauvais encodage / PDF image errors="ignore" deja gere ; un PDF scanne sans texte est normal -> l'ignorer
chunks enormes ou minuscules CHUNK_SIZE mal regle revenir a 800 / 120

Etape 2 (R2) - Embeddings + vector store : transformer en nombres et ranger

Ce que vous faites : transformer chaque chunk en vecteur (une liste de 384 nombres) puis le ranger dans le vector store pour pouvoir le rechercher par le sens.

Pourquoi ? Un ordinateur ne compare pas des phrases, il compare des nombres. Un embedding traduit le sens d'un texte en nombres : deux textes de sens proche donnent deux vecteurs proches. C'est ce qui permet de retrouver "lutte anti-blanchiment" quand on cherche "blanchiment".

Analogie : on donne a chaque morceau une coordonnee GPS dans un espace du sens. Les morceaux qui parlent de la meme chose sont voisins sur la carte. Chercher = trouver les voisins de la question.

Comment s'y prendre

  1. app/embed.py est deja ecrit : il lit chunks.jsonl, calcule les vecteurs (modele local all-MiniLM-L6-v2, telecharge au 1er run) et appelle votre vector store.
  2. Votre vrai travail : implementer l'adaptateur de votre vector store dans app/store.py.
  • Projet A (Qdrant) : QdrantStore est deja implemente -> il vous sert d'exemple, verifiez juste qu'il tourne.
  • Projet B (Chroma) : implementer ChromaStore (voir indices projet B).
  • Projet C (pgvector) : implementer PgVectorStore (voir indices projet C).

Un adaptateur a 3 methodes : ensure_collection (creer la "table" si absente), upsert (inserer), search (chercher les top-k). Copiez la logique de QdrantStore comme modele.

🟨 Warning - le piege n°1 ici : la dimension. Le modele produit des vecteurs de 384 nombres. Si vous declarez une autre taille dans le vector store, l'insertion echoue. Gardez EMBEDDING_DIM=384.

Comment verifier

docker compose run --rm api python -m app.embed

Output attendu :

[embed] 64/612 chunks indexes
[embed] ...
[embed] Termine : 612 chunks dans le vector store (all-MiniLM-L6-v2, dim=384)

Test rapide de recherche (sans LLM) :

docker compose run --rm api python -m app.retrieve "mesures de securite"
# doit afficher quelques chunks avec un score (ex. [0.81] cnil_guide... :: ...)

🟢 OK si : la commande affiche des chunks pertinents avec un score entre 0 et 1, le plus pertinent en premier.

Erreurs frequentes :

🟨 Symptome Cause Correction
NotImplementedError adaptateur store pas encore code implementer ensure_collection/upsert/search
erreur de dimension a l'upsert dim != 384 EMBEDDING_DIM=384, taille de collection = 384
collection vide apres embed collection recreee a chaque run tester l'existence avant de creer (idempotent)
1er run tres long telechargement du modele (~90 Mo) normal, c'est mis en cache ensuite

Etape 3 (R3) - Retrieval + generation : chercher et faire rediger le LLM

Ce que vous faites : pour une question, retrouver les top-k chunks, les donner au LLM avec une consigne stricte, et renvoyer la reponse avec ses sources.

Pourquoi une consigne stricte ? Sans garde-fou, le LLM invente (hallucination). On l'oblige a repondre uniquement depuis le contexte fourni et a refuser s'il ne trouve pas (cf. l'incident d'hallucination dans le corpus seed).

Comment s'y prendre

  1. app/retrieve.py : transforme la question en vecteur et cherche les top-k. Deja amorce. Ajout important (surtout projet A) : un seuil -> si le meilleur score est trop bas, on refuse.
  2. app/generate.py : construit le prompt et appelle le LLM. Le prompt systeme anti-hallucination est deja fourni -> lisez-le, comprenez-le, ajustez-le.
  3. app/api.py : expose POST /ask. Deja amorce.

Indice (le prompt qui force la citation, deja dans generate.py) :

SYSTEM_PROMPT = (
    "Vous repondez UNIQUEMENT a partir du contexte fourni. "
    "Citez vos sources [source: <document>]. "
    "Si le contexte ne contient pas la reponse, repondez : "
    "'Je ne dispose pas de cette information dans le corpus.'"
)

Comment verifier

docker compose up -d
curl -X POST http://localhost:8000/ask \
  -H "Content-Type: application/json" \
  -d '{"question": "Quelles mesures de securite sont recommandees ?"}'

Output attendu (la forme) :

{
  "answer": "Les mesures recommandees incluent ... [source: cnil_guide...]",
  "sources": [{"doc": "cnil_guide...", "chunk_id": "...", "score": 0.81}],
  "latency_ms": 1840,
  "tokens": {"prompt": 1203, "completion": 187}
}

Testez aussi une question hors corpus ("Quelle est la capitale de l'Australie ?") : la reponse doit etre le refus. Si le LLM invente, votre garde-fou ne marche pas.

🟢 OK si : question dans le corpus -> reponse + sources ; question hors corpus -> refus.

Erreurs frequentes :

🟨 Symptome Cause Correction
erreur 401 / cle invalide GROQ_API_KEY absente ou fausse verifier .env, regenerer la cle
erreur 429 rate limit free tier espacer les requetes, ne pas tester en rafale
le LLM invente prompt pas respecte / pas de seuil renforcer le prompt + seuil de similarite
reponse sans sources sources non transmises dans le prompt verifier build_prompt (les [source: ...])

Etape 4 (R4) - API conteneurisee + mesures

Ce que vous faites : verifier que tout demarre en une commande, et mesurer votre pipeline.

Pourquoi mesurer ? Parce qu'en production on ne dit pas "c'est rapide", on dit "latence p95 = 2,1 s, cout = 0,002 USD / question". C'est ce qui distingue un prototype d'un service (cf. FinOps J1).

Comment s'y prendre

  1. Verifier votre docker-compose.yml : docker compose up depuis un clone propre doit tout monter.
  2. Completer app/metrics.py : au moins 1 metrique qualite + 1 metrique exploitation.
  • Qualite : score de similarite moyen, taux de refus.
  • Exploitation : latence p50/p95, tokens, cout projete. 3. (Bonus selon projet) : analytique (B), monitoring Grafana (C), golden dataset (A).

Indice - mesurer la latence et le taux de refus sur un lot :

import time
from app.generate import answer
from app.metrics import latency_stats, refusal_rate

questions = ["...", "...", "..."]   # votre jeu de test
lat, reps = [], []
for q in questions:
    t = time.perf_counter()
    r = answer(q)
    lat.append((time.perf_counter() - t) * 1000)
    reps.append(r["answer"])
print(latency_stats(lat))     # {'p50': ..., 'p95': ...}
print("taux de refus:", refusal_rate(reps))

🟢 OK si : un clone tout neuf demarre et repond. C'est le test exact de la correction.
